- (decision) AD paradigm: layered architecture inside a unified Next.js App Router app with a repository (port) boundary. Layers: UI (RSC/client components) -> API (Route Handlers) -> domain/service -> repository -> Postgres. [ADOPTED]
- (decision) AD dependency direction: deps flow UI->API->service->repository->DB, never backward; UI never imports the DB client or repository directly (only via API/fetch). Prevents layering violations + circular deps.
- (decision) AD single data-access repository: all persistence behind one Todo repository interface; no SQL/DB client anywhere else. Prevents scattered SQL / two owners of the store / hardcoded DB coupling; enables swap + test. [ADOPTED]
- (decision) AD Postgres system-of-record [ADOPTED by user]: durable containerized Postgres, server-side access only via repository. Owns the todos table.
- (decision) AD API contract: REST Route Handlers under app/api/todos (collection) + app/api/todos/[id] (item). GET list, POST create, PATCH edit/toggle, DELETE. Consistent JSON envelope + status codes + { error } shape. Prevents divergent endpoint/response shapes.
- (decision) AD shared Todo schema single source of truth: one TS type + Zod schema imported by client, API, repository; validated at the API boundary. Prevents client/server shape drift.
- (decision) AD validation server-authoritative (FR-9): trim, reject empty/whitespace, cap 1000 code points enforced in the handler via the shared schema; client mirrors for UX only. Prevents inconsistent/only-client validation.
- (decision) AD state-mutation paradigm = optimistic update + rollback (FR-10/11): client updates immediately, reconciles with server, rolls back per-op on failure; tasks without a server id are non-mutable until create resolves. Prevents divergent client update strategies.
- (decision) AD voice pack: all user-facing copy in one centralized module keyed -> array of variants; UI never hardcodes strings. Voice-scope boundary: API field names, error codes, logs, code identifiers stay plain (no Torgue). Binds FR-14..20.
- (decision) AD deterministic rotation selector: pure function (key, lastShownIndex, rng)->variant with injected rng; never Math.random() in components; no back-to-back repeat; all variants reachable. Binds FR-16/17. Prevents untestable rotation.
- (decision) AD accessibility contract: ALL-CAPS via CSS text-transform (never stored caps); stable descriptive aria-name/role on controls, toasts, confirm/cancel dialog even as visible label rotates; rotation never alters state-bearing text (sort selection, on/off). Target WCAG 2.2 AA, zero critical. Binds FR-17/20, NFR-4.
- (decision) AD XSS-safe rendering: task text stored raw, rendered only through React auto-escaping; dangerouslySetInnerHTML forbidden for user/task content. Binds security review, FR-1/3.
- (decision) AD deployment = Docker Compose: multi-stage Dockerfile for the Next.js app (non-root user, HEALTHCHECK); docker-compose orchestrates app + postgres with named volume, per-service healthchecks, env via .env + compose profiles (dev/test). Health endpoint GET /api/health verifies DB connectivity. Binds ops/deploy.
- (decision) AD test strategy: Vitest unit/integration (repository vs test DB, route handlers, rotation selector, validation schema); Playwright E2E >=5 (create, edit, toggle-complete, delete-with-confirm, empty state, error/rollback); axe-core a11y via Playwright; >=70% meaningful coverage. Binds NFR + QA success criteria.
