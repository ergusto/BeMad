## What went well

- **The adversarial 3-layer review earned its keep every single story.** It caught issues that unit + E2E green would have hidden: the **edit-not-actually-optimistic AC gap** (2.2, Acceptance Auditor), the **`delete/rollback` stale-index order corruption** under concurrent deletes (2.2, Blind+Edge), the **"oldest" test that passed on a no-op** (2.3), and the **tautological overflow assertion + sub-WCAG checkbox target** (2.4). None were caught by the passing test suite alone.
- **The AD-14 "pure core" pattern kept everything testable without a DOM harness.** The reducer (2.1/2.2) and `sortEntries` (2.3) are pure functions with rich unit suites; optimistic and sort concerns stayed cleanly separated because sort is a *derived view* over canonical `entries` — which is exactly why 2.3 didn't disturb 2.2's rollback logic.
- **Network-interception E2E scaled to every new concern deterministically** — optimistic reconcile/rollback (holding responses to observe pending state), sort reordering, and responsive layout — all DB-independent via `page.route`.
- **Per-story commits happened** (Epic 1 action item cleared): three clean commits on `feat/epic-2-optimistic-store`.

## What was harder / lessons

- **Cross-cutting ACs need per-operation decomposition.** "Every mutation updates optimistically ≤100ms" was satisfied for create/toggle/delete but the first dev pass left **edit** awaiting the server before closing the editor. The AC was true in aggregate but false for one operation. **Lesson:** when an AC says "all X", enumerate each X in the dev checklist and test each.
- **Tests can pass on a no-op.** 2.3's "oldest" fixtures were already createdAt-ascending (expected == input), and 2.4's page-overflow assertion was clamped by `overflow-x:hidden` on `body`. Both would have passed even if the feature did nothing. **Lesson:** for derived/presentational behavior, make the test's expected result differ from the trivial/identity baseline (shuffle inputs; assert on an un-clipped element).
- **Still no component-test harness (RTL).** UI failure-paths keep pushing to E2E. It didn't hurt this epic (E2E covered them) but remains a recurring deferral that will matter more for Epic 3's voice-rendering logic.

## 🚨 Significant discovery — flag before Epic 3

**Epic 3's rotating voiced copy collides with the accessible names the entire E2E suite depends on.** All 50 E2E locate controls by stable accessible names — `Add task`, `New task`, `Sort tasks`, `Completed: <text>`, `Edit`, `Save`, `Cancel`, `Delete: <text>`, `Confirm delete: <text>`, `Retry`, `Dismiss`. Story 3.3 ("wire all surfaces to the voice pack") replaces hardcoded user-facing strings with rotating variants. If those rotating strings become the controls' **accessible names**, (a) every locator breaks and the suite destabilizes, and (b) screen-reader names would rotate — a real a11y regression, and a direct tension with **FR-17** (rotation must not alter text conveying state/selection) and **AD-10** (stable aria names/roles).

**Recommendation (prep before Story 3.3):** confirm the strategy — voiced copy in *visible* text while **accessible names / `aria-label`s stay stable and plain** (which AD-10 already mandates), or migrate E2E control locators to `data-testid`/role+position. Decide this in Epic 3 planning; it affects how 3.3 is built and tested. Good news: 2.3 already kept `SORT_OPTIONS` labels as data and errors are plain strings in the store/client, so the voice layer has clean seams to plug into.
