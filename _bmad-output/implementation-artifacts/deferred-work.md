## Deferred from: code review of story-2.2 (2026-07-07)

- **Timeout/abort after a server-side success causes a false rollback** — every request is now bounded by `AbortSignal.timeout(10_000)` (`lib/todos-client.ts`). If the server commits a create/update/delete but the response is slow (or the connection drops) past the timeout, the client treats the abort as an ordinary failure and rolls back state the server actually kept. Worst case: a create times out → optimistic row removed + text retained → the user's natural retry issues a second POST → **duplicate todo** on next reload. Toggle/edit/delete diverge silently from the server until reload. Inherent optimistic + timeout tradeoff; a proper fix needs request idempotency keys and/or a reconcile-on-reload — both out of Story 2.2 scope (no API/route/schema changes allowed). Also note the timeout now bounds the GET/load path: a legitimately slow (>10s) initial load becomes an error state (with Retry) where it previously waited indefinitely — judged acceptable (better than an infinite spinner). Revisit alongside a future reconciliation/idempotency story. [lib/todos-client.ts, lib/store/todo-store.tsx]

## Deferred from: code review of story-3.3 (2026-07-08)

- **[Story 3.4 TOP ITEM] Rotating controls lost per-task accessible-name context** — Story 3.3 adopted the retro decision "voiced visible text IS the accessible name" (WCAG 2.5.3) and removed the per-row `aria-label={`Delete: ${task}`}` / `Edit`. A screen-reader user navigating by button now hears "OBLITERATE, OBLITERATE, …" with no task context, and the name rotates. The addendum's a11y note wanted a *stable, descriptive* aria-label, which conflicts with 2.5.3 when the visible text is voiced. Story 3.4 (Accessibility of rotating copy, WCAG 2.2 AA) must resolve this holistically — options: accessible name = `${voicedVisible}: ${task}` (contains the visible label → 2.5.3-safe, adds task context), or a visually-hidden per-row task reference, or `aria-describedby`. Audit with @axe-core/playwright. [app/todo-app.tsx TodoItem edit/delete buttons]
- **Variant-0 first-paint flash on interaction-only-mounted voiced surfaces** — `useVoice` renders variant 0 then rotates in a post-hydration mount effect (correct + necessary for SSR'd surfaces like loading/empty). But surfaces that only ever mount client-side after an interaction (EditRow Save/Cancel, DeleteConfirm prompt/confirm/cancel, the error banners) visibly paint variant 0 then swap one frame later. A provider enhancement could seed the index synchronously for non-SSR mounts (e.g. a lazy `useState(() => controller.pick(...))` path) while keeping the effect for SSR'd surfaces. Cosmetic; revisit in 3.4 polish. [lib/voice/provider.tsx, app/todo-app.tsx]
- **Client validation misclassifies non-empty/non-too-long schema failures** — the UI picks the voiced validation key with a binary `[...text.trim()].length > MAX` check, so a rare input that is non-empty and ≤1000 code points but still rejected (e.g. contains a NUL, `todoTextSchema`) shows the "empty task" message. Obscure paste edge; would need a third voiced key ("invalid task"). Low. [app/todo-app.tsx handleSubmit/save]

## Deferred from: code review of story-3.4 (2026-07-08)

- **Focus falls to `<body>` after a successful (or failed) delete** — Story 3.4 added focus-return to the Delete trigger on the *cancel* path (the Epic 1 retro item).
